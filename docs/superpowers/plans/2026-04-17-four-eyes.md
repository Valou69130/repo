# Four-Eyes Approvals Implementation Plan

**Goal:** Enforce a second-user approval gate on high-value margin-call acceptances. Same-user self-approval rejected at the service layer.

**Scope decision:** Four-eyes is implemented on `accept` only this phase. Issue-from-draft and cancel keep their existing immediate-transition flow тАФ widening the scope requires state-machine changes that belong in a later phase.

**Architecture:** A new `src/routes/approvals.js` exposes the approver-facing endpoints. `marginCalls.js` accept route is modified so that when `call_amount > agreement.four_eyes_threshold` the call transitions to `pending_four_eyes`, writes an `approvals` row with `entity_type='margin_call_accept'`, and returns `202` with `approvalPending` instead of `200`. The grant endpoint appends `four_eyes_granted` тЖТ `agreed`; reject appends `four_eyes_rejected` тЖТ `disputed`.

**Tech Stack:** Express 4, better-sqlite3, node:test + supertest.

## Task 2 тАФ `src/routes/approvals.js`

- `GET /pending` тАФ `SELECT * FROM approvals WHERE status='pending' AND requested_by_user_id != ?` (current user)
- `POST /:id/grant` тАФ permission `canApproveFourEyes`, refuses self-approval, appends `four_eyes_granted` on the entity_id margin call, updates approval row
- `POST /:id/reject` тАФ same guard rails, appends `four_eyes_rejected`

## Task 3 тАФ accept gate in `marginCalls.js`

If `call_amount > agreement.four_eyes_threshold`:
- appendEvent `four_eyes_requested` тЖТ `pending_four_eyes`
- INSERT INTO approvals (entity_type='margin_call_accept', entity_id=call.id, requested_by_user_id, status='pending')
- Return 202 `{ marginCall, event, approvalPending: { id, requiredRole: 'Credit Approver' } }`

Else: unchanged fall-through to existing `accepted` event.

## Task 4 тАФ wire router

Add `app.use('/approvals', require('./routes/approvals'))` in `src/index.js`.
